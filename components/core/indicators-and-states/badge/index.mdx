# Badge

## Overview

Badge is a small pill-shaped label used to communicate status, category, or contextual metadata at a glance. It is a display-only component with no interactive states.

The Badge component exists in two sets:

- **Badge/Main** — six semantic color types. Each type carries a distinct meaning (error, info, success, warning, default, outlined). Use when the badge communicates system status or state.
- **Badge/Utility** — five non-semantic identity colors (purple, violet, cyan, teal, pink). Use when the badge identifies a category, entity type, or label where color is decorative rather than status-driven.

Both sets share the same shape, size, typography, and anatomy. They differ only in their color tokens and intended purpose.

## Anatomy

Badge is composed of the following parts:

| Part | Required | Description |
|---|---|---|
| **Container** | Yes | Pill-shaped surface (border-radius: 9999px). Background and border determined by type. Padding: 8px horizontal, 4px vertical. |
| **Icon** | Optional | Leading icon, 16px. Controlled by `iconShow` (default: `true`) and `iconType` (default: alert-war-triangle). Decorative — must be `aria-hidden`. |
| **Label** | Yes | Short text string. Manrope SemiBold, 12px, `whitespace-nowrap`. Color determined by type. |

- The label is always required. Do not render a Badge without visible text.
- The label must not wrap — keep it short (1–3 words).
- The icon is present by default (`iconShow=true`) and uses the `Customer-Icon/alert-war-triangle` icon per type. Pass `iconShow={false}` to hide it or `iconType` to override the icon node.
- For the dedicated partner variant on Venue cards with fixed icons and labels, see [Badge Partner](/components/ds-customer/indicators-and-states/badge-partner/index).

## Variants

### Badge/Main

Six types for communicating system status or state. Each type carries a defined semantic meaning — do not choose types based on visual appearance.

| Type | Color | Semantics |
|---|---|---|
| `default` | Neutral gray | Generic label, category, or metadata. No status meaning. |
| `outlined` | No fill, bordered | De-emphasized or inactive label. |
| `error` | Red | Error, critical status, failed state, blocked action. |
| `info` | Blue | Informational status, pending state, neutral system metadata. |
| `Success` | Green | Completed, confirmed, active, or approved state. |
| `Warning` | Yellow | Cautionary state, attention required, expiring or degraded. |

> Choose the type based on the **semantic meaning** of the status — not its visual appearance. Do not use `error` for warnings, or `warning` for errors.

### Badge/Utility

Five identity colors for labeling categories, entity types, or contextual properties where color carries no status meaning. Colors are drawn from the `chart-pair` token collection.

| Type | Color | Use for |
|---|---|---|
| `purple` | Purple | Category or entity identification. |
| `violet` | Violet | Category or entity identification. |
| `cyan` | Cyan | Category or entity identification. |
| `teal` | Teal | Category or entity identification. |
| `pink` | Pink | Category or entity identification. |

> Utility badge colors are decorative and carry no semantic meaning. Do not use them to communicate system status. Assign colors consistently within a product context — the same type should always map to the same category.

## Tokens & specs

### Supported variants

- **Badge/Main:** six types — `default`, `outlined`, `error`, `info`, `Success`, `Warning`
- **Badge/Utility:** five types — `purple`, `violet`, `cyan`, `teal`, `pink`

### Supported sizes

Single size only — no size variants.

| Property | Token | Value |
|---|---|---|
| Padding horizontal | `--spacing-8` | 8px |
| Padding vertical | `--spacing-4` | 4px |
| Border radius | — | 9999px (full pill) |

### Supported states

No interactive states confirmed. Display-only.

## Usage guidelines

- Use Badge to communicate the **status, category, or type** of an adjacent element — not as a primary action or navigation element.
- Place Badge near the element it describes: next to a title, alongside a list row, or within a card header.
- Use a **single Badge per item** unless each badge communicates a genuinely distinct dimension (e.g., one for status, one for category).
- Use **Badge/Main** when the badge communicates a system state or outcome. Choose `default` or `outlined` for neutral labels; reserve `error`, `info`, `Success`, and `Warning` for genuine status meaning.
- Use **Badge/Utility** when the badge identifies a category, entity type, or label where color is decorative. Assign colors consistently — the same type should always map to the same category within a given screen or product.
- Do not mix Badge/Main and Badge/Utility types to communicate the same kind of information on the same item.
- Do not use Badge as a notification counter. Use a dedicated counter or dot component for that pattern.
- Keep labels short — 1–3 words. `whitespace-nowrap` is applied; longer labels will overflow.

## Accessibility

- Badge text is informational. Screen readers will read it as part of the surrounding content.
- Do not rely on color alone to communicate status — the label text must be self-explanatory regardless of the type color. This is especially important for Badge/Utility, where colors carry no inherent meaning.
- If a Badge appears without adjacent explanatory text, consider adding a visually hidden label or `aria-label` on the parent element to provide full context.

## Do / Don't

**Do:**
- Use the type that matches the semantic meaning of the status.
- Keep labels short (1–3 words, sentence case).
- Use `default` or `outlined` for neutral, non-status labels.
- Ensure the label text communicates meaning independently of color.

**Don't:**
- Don't use `error` for warnings or `warning` for errors — each Badge/Main color has a defined meaning.
- Don't use Badge/Utility types to communicate system status — they are for identity and category labels only.
- Don't assign Badge/Utility colors inconsistently — the same color should always represent the same category.
- Don't write long labels — `whitespace-nowrap` is applied.
- Don't use Badge as a navigation control or interactive filter unless a confirmed interactive variant exists.
- Don't add icons unless a confirmed icon variant is defined.
- Don't use Badge as a notification counter.
