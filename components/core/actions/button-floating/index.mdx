# Button Floating

## Overview

Button Floating is a circular icon button that appears elevated above a content surface — such as a map, image, or layered view. It uses a white neutral background and a drop shadow (`elevation/light/01`) to create the visual impression of floating above the content beneath it.

Unlike [Button Icon](/components/button-icon), it has no type variants and no visible label. It is reserved for a single, contextual action directly related to the surface it floats over.

Because it has no visible label, an `aria-label` is required on every instance.

## Anatomy

The Button Floating is composed of two parts:

| Part | Required | Description |
|---|---|---|
| **Container** | Yes | Square wrapper with equal width and height. `rounded-full` border-radius renders it as a perfect circle. White neutral background (`--surface/neutral/sunken`). Drop shadow (`elevation/light/01`) creates the floating appearance. |
| **Icon** | Yes | A single centered icon (24×24px for `md`). The only visible content. No label. |

## Variants

Button Floating has a **single visual style**. There are no type variants.

The white neutral surface with elevation shadow is the defining visual characteristic. This distinguishes it from [Button Icon](/components/button-icon), which supports six types with different fill colors and hierarchies.

## States

Button Floating supports four states. There is no hover state — this component is designed primarily for touch and mobile contexts.

| State | Trigger |
|---|---|
| `Default` | Resting state, no interaction. |
| `Pressed` | Button is actively tapped or clicked. |
| `Disabled` | Action is unavailable. Non-interactive. |
| `Focus` | Button receives keyboard focus. |

## Usage guidelines

- Use Button Floating only for a **single, contextual action** directly related to the surface it floats above — for example, re-centering a map or locking a GPS position.
- Place only **one floating button per surface**. Avoid stacking multiple floating buttons in the same area.
- Position the button at a **consistent, predictable location** within the surface — typically a fixed corner. TODO: Confirm the canonical corner position within the DoJo product.
- Do not use Button Floating for primary navigation or app-level actions. Use the labeled [Button](/components/button) for those.
- Do not use Button Floating when space allows for a labeled button. Labels always improve clarity.
- Provide a tooltip matching the `aria-label` to aid discoverability for sighted users.

## Accessibility

- `aria-label` is **required** on every instance. The button has no visible label. Without it, the button has no accessible name.
- The icon element should be `aria-hidden="true"` so screen readers rely on the button's `aria-label`.
- The `Focus` state is designed and present. Do not suppress or override the focus ring.
- Implement `disabled` or `aria-disabled="true"` for disabled instances. Visual state alone is not sufficient.
- Provide a tooltip that matches the `aria-label` text for sighted users who may not recognize the icon.

## Do / Don't

**Do:**
- Always provide an `aria-label` that describes the action (e.g., `aria-label="Center map"`).
- Use for a single, surface-specific contextual action.
- Position consistently and predictably within the content surface.
- Provide a tooltip to expose the action's purpose on hover and focus.

**Don't:**
- Don't place more than one floating button on the same surface without a clear hierarchy reason.
- Don't use Button Floating without an `aria-label`.
- Don't use it for global app-level actions — it is surface-specific.
- Don't remove the elevation shadow — it is part of the component's visual identity.
- Don't use it when a labeled button fits in the available layout.
