# Dropdown Input

## Overview

Dropdown Input is a form field wrapper that combines an optional label, a tappable pill-shaped select button (`select_button`), and optional annotation text. It does not contain the picker or overlay itself — it acts as the visible trigger and label container that opens a separate selection UI (such as a date picker, number picker, or bottom sheet).

It is used across the Dojo apps wherever a user must select a value from a predefined set via a picker or modal panel.

## Anatomy

The Dropdown Input is composed of three stacked elements:

1. **Label** *(optional)* — a short text label displayed above the select button. Identifies the field.
2. **SelectButton** — the pill-shaped interactive trigger. Contains:
   - **Left Icon** *(optional)* — a 16px contextual icon, left-aligned inside the button
   - **Value / Placeholder Text** — shows the selected value, or a placeholder string when empty
   - **Chevron Icon** — a fixed 24px dropdown chevron on the right side, always visible
3. **Annotation** *(optional)* — a helper text row below the select button, prefixed with a 16px info icon

## Variants

The Dropdown Input itself has no visual type variants. Variation is expressed through its props:

- **With left icon** — the select button includes a 16px icon to the left of the value text
- **Without left icon** — the select button shows only the text and chevron

The select button (`select_button`) internally supports multiple named types (e.g. `preselected`). Full type enumeration is defined in the base component.

## States

| State | Description |
|-------|-------------|
| **Placeholder** | No value selected. Placeholder text is shown in `--text/placeholder` (muted color). |
| **Preselected / Filled** | A value has been selected. Value text is shown in `--button/secondary/text-default` (body color). |

## Usage guidelines

- Use Dropdown Input when the user must select a value from a set that requires a separate picker UI (e.g. date, time, number, category).
- Do not use Dropdown Input for selections that can be shown inline without an overlay — use a segmented control or radio group instead.
- Always pair with a label when the field appears in a form. The label may be omitted only when the placeholder text alone is sufficient to identify the field.
- Use annotation text to provide additional context, constraints, or helper guidance — not to repeat the label.
- Place Dropdown Input fields vertically stacked in forms, full-width within their container.
- The select button opens a picker, bottom sheet, or overlay. The Dropdown Input component itself does not manage this interaction — the parent screen or form is responsible for handling the tap event and opening the appropriate UI.

## Accessibility

- The select button must have an accessible label derived from the visible label text or the placeholder.
- The chevron icon is decorative and should be hidden from assistive technology (`aria-hidden`).
- The left icon is decorative in most cases. If it carries semantic meaning, provide an `aria-label` or visually hidden text.
- When a value is selected, the accessible name of the button should reflect the current value, not the placeholder.
- The annotation text should be associated with the select button via `aria-describedby`.

## Do / Don't

**Do:**
- Use a clear, short label (2–4 words) that describes what the user is selecting
- Show a meaningful placeholder that hints at the expected value (e.g. "Select dates…", "Select vehicle type")
- Use annotation text to provide constraints or examples relevant to the field

**Don't:**
- Use Dropdown Input for free-form text input — use a text input field instead
- Stack left icons inconsistently across fields in the same form
- Use the annotation slot for error messages — error states require a dedicated pattern
