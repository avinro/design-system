# Alert Dialog

## Overview

Alert Dialog is a full-screen modal overlay that stops the user in place and forces an explicit decision before they can continue. It is intentionally constrained — short copy, a visible icon, and a maximum of two actions — because it is reserved for situations where proceeding without a clear response could cause harm, data loss, or significant confusion.

Alert Dialog is **not** a general-purpose modal. Use it only when the stakes are high enough to justify interrupting the user's flow entirely. For forms, lightweight confirmations, or non-urgent messages, use a different component from the messaging family.

## Anatomy

The component has two layers:

1. **Overlay** — a semi-transparent full-screen backdrop (`--surface/neutral/overlay`) that dims the content behind and signals that the UI is blocked.
2. **Dialog card** — a rounded sheet anchored to the bottom of the screen. Contains four internal sections stacked vertically:
   - **Icon container** — an 80×80px filled circle with a type-specific background color and a 40×40px icon that signals the intent.
   - **Content** — centre-aligned title and body text.
   - **Actions** — a vertical stack of one or two full-width pill buttons.

## Variants

Four intent types are supported. Each changes the icon, the icon background color, and the styling of the second action button:

| Type | Icon | Icon background | When to use |
|------|------|-----------------|-------------|
| `Warning` | Triangle warning | `--notification/warning--bg` | Preventable mistakes; user must choose before proceeding |
| `Error` | Circle exclamation | `--notification/error-bg` | Blocking failures that cannot be bypassed without acknowledgement |
| `Destructive` | Trash / delete | `--notification/error-bg` | Irreversible actions — delete, cancel booking, remove payment method |
| `Info` | Info circle | `--notification/info-bg` | Neutral interruptions requiring acknowledgement before continuing |

All four types share the same card layout. The second button differs:

| Type | Primary button | Second button |
|------|----------------|---------------|
| `Warning` | Primary (neutral dark) | Secondary (neutral) — or omitted for acknowledge-only |
| `Error` | Primary (neutral dark) | Secondary (neutral) |
| `Destructive` | Primary (neutral dark) — the safe exit | Destructive (red) — the irreversible action |
| `Info` | Primary (neutral dark) | Secondary (neutral) — or omitted for acknowledge-only |

{/* ![Alert Dialog variants](./variants.png) */}

## States

Alert Dialog itself has no interactive states beyond open/closed. Button states (hover, pressed, disabled) follow the core Button component specification.

The `primaryAction` prop controls whether one or two buttons are rendered. Set it to `false` for acknowledge-only dialogs where there is no meaningful second choice.

## Usage guidelines

### When to use Alert Dialog

- **Destructive confirmation** — deleting an item, cancelling a booking or ride, removing a payment method.
- **Critical mismatch** — when the user's inputs or context conflict in a way that could cause a bad outcome (e.g. wrong pickup address).
- **Blocking error** — a failure that cannot be bypassed without explicit acknowledgement.
- **High-risk warning** — the user must explicitly choose the next step before the system can proceed.

### When NOT to use Alert Dialog

| Situation | Use instead |
|-----------|-------------|
| Forms or tasks requiring multiple inputs | Standard Dialog |
| Routine success confirmation ("Saved!") | Toast |
| Lightweight recoverable issue (Undo / Retry) | Snackbar |
| Non-urgent informational content | Toast or inline message |

### Actions

- Use **1 action** for acknowledge-only dialogs (`Info` or `Warning` where no real choice exists).
- Use **2 actions** for confirmation or safe-vs-risky decisions. Maximum 2.
- Never add a third button.
- Place the **safe or recommended action** as the primary button.
- Place the **destructive or irreversible action** as the second button using destructive button styling.
- Do **not** use destructive button styling on the safe action.

### Dismiss behavior

Choose one governance rule and apply it consistently across the product:

- **Strict (recommended):** No close icon. The user must choose one of the provided actions.
- **Permissive:** A close icon or dismiss gesture is allowed on `Info` and `Warning` types only. Never allow dismiss on `Destructive` or `Error`.

### Content rules

- **Title:** short, explicit. Prefer a question format when asking for a decision (e.g. "Cancel this ride?").
- **Body:** 1–3 lines maximum. No paragraphs. No inline links.
- **Button labels:** use explicit, consequence-describing labels. Avoid generic labels.
  - ✅ "Cancel booking", "Delete plan", "Remove card"
  - ❌ "Yes", "Confirm", "Proceed"

## Accessibility

- The dialog must have `role="alertdialog"`.
- `aria-labelledby` must point to the title element.
- `aria-describedby` must point to the body text element.
- On open, focus must move to the first action button (or the dialog container).
- Focus must be trapped within the dialog while it is open.
- The overlay must not be interactable or focusable.
- `Escape` key behavior depends on the dismiss governance chosen: strict dialogs should not respond to `Escape`; permissive dialogs may allow it for `Info`/`Warning` types only.
- All button labels must be descriptive enough to communicate consequence without surrounding context.

## Do / Don't

**Do:**
- Use for genuinely high-stakes interruptions — destructive actions, blocking errors, critical mismatches
- Write the title as a direct question when the user must make a choice
- Label buttons with the exact consequence: "Cancel booking", "Delete account"
- Put the safe action first (primary button), the risky action second
- Restrict to 2 actions maximum

**Don't:**
- Use Alert Dialog for informational messages that do not require a decision
- Use generic button labels like "OK", "Yes", "Confirm" when consequences are real
- Include form fields, dropdowns, or multi-step content inside the dialog
- Apply destructive button styling to the safe/cancel action
- Show more than two action buttons
